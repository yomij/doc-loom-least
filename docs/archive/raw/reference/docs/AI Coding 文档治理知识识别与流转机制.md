# AI Coding 文档治理中的知识识别、流转与 Agent 实现机制

## 1. 背景与核心问题

在 AI Coding 场景下，知识不仅存在于正式文档中，也会不断产生于需求讨论、技术方案、代码实现、PR Review、测试用例、Bug 修复、Agent 对话记录等过程中。

如果这些知识没有被治理，就会出现以下问题：

- 关键规则只停留在聊天记录或个人经验中；
- AI Agent 生成代码时缺少必要上下文；
- 同一业务概念在不同文档中有不同说法；
- 旧文档没有及时废弃，误导后续开发；
- 需求实现中产生的经验无法沉淀为组织级知识资产。

因此，AI Coding 文档治理的核心不是“把知识写进文档”，而是建立一套知识从临时经验到稳定规范的流转机制。

可以将其理解为一条知识供应链：

```text
需求 / 对话 / Issue / PR / Code Diff / 测试 / Review
  ↓
知识识别
  ↓
知识归纳
  ↓
知识分类
  ↓
知识卡片化
  ↓
知识评审
  ↓
晋升 / 保留 / 降级 / 废弃
  ↓
进入对应文档层级
  ↓
反哺后续 AI Coding
```

---

## 2. 文档层级设计

不同文档层级应该承载不同稳定性、不同影响范围的知识。

| 层级 | 文档类型 | 承载的知识 | 特点 |
|---|---|---|---|
| L0 原则层 | 工程原则、架构原则、编码规范、AI Coding 规范 | 长期稳定的原则、红线、强约束 | 变化最少，影响最大 |
| L1 架构层 | 系统架构、领域模型、模块边界、技术选型 | 跨系统、跨模块的结构性知识 | 需要评审后变更 |
| L2 能力层 | 业务能力说明、通用组件说明、接口契约、领域规则 | 可复用的中层知识 | 多需求共享 |
| L3 需求层 | PRD、技术方案、需求设计、验收标准 | 某个需求的上下文、方案、决策 | 随需求变化 |
| L4 实现层 | 代码注释、README、测试说明、运行手册、PR 说明 | 具体实现、局部经验、排障信息 | 最贴近代码 |
| L5 临时层 | Chat 记录、Issue、Commit、Debug 日志、Agent 中间结论 | 未验证、碎片化、临时知识 | 不直接作为正式文档 |

治理目标不是让所有知识都进入高层文档，而是让知识进入“合适”的层级。

---

## 3. 知识在文档层级之间的流转方式

知识流转主要分为三类：

1. 上行晋升；
2. 下行实例化；
3. 横向传播。

---

## 4. 上行晋升：从局部经验到通用知识

上行晋升是指知识从低层级文档进入更高层级文档。

典型路径：

```text
Chat / Issue / PR / Code Diff
  ↓
需求实现说明
  ↓
模块设计说明
  ↓
架构规范 / 工程规范
```

### 4.1 示例

某个需求中发现：

```text
订单状态变更必须通过状态机，不允许直接更新 status 字段。
```

它最初可能只是 PR Review 中的一条评论。

如果只适用于当前需求，可以保留在 L3 需求文档或 L4 实现说明中。

如果多个订单相关需求都需要遵守，应晋升到 L2 的“订单领域规则”。

如果它体现了一种全局建模原则，例如“核心状态流转必须通过领域服务封装”，则可以进一步晋升到 L0 或 L1。

### 4.2 晋升判断标准

| 判断问题 | 说明 |
|---|---|
| 是否被多个需求复用？ | 出现 2 次以上就应考虑晋升 |
| 是否影响多个模块？ | 跨模块知识不能只留在局部文档 |
| 是否是约束而不是描述？ | 约束、规则、接口契约更值得晋升 |
| 是否已经被代码和测试验证？ | 未验证知识不能直接晋升为规范 |
| 是否会影响后续 AI Coding？ | 会影响 Agent 生成代码的知识，优先晋升 |

---

## 5. 下行实例化：从高层规则到具体需求

下行流转是指高层文档中的原则、架构、规范，在具体需求中被引用和实例化。

### 示例

L0 工程规范中定义：

```text
所有外部系统调用必须具备超时、重试、幂等保护。
```

在某个支付需求的 L3 技术方案中，应实例化为：

```text
本需求调用支付网关时，遵循《外部系统调用规范》，具体实现为：

- 超时时间：3 秒；
- 最多重试：2 次；
- 幂等键：payment_request_id。
```

下行实例化可以保证 AI Coding Agent 在实现具体需求时，不只读取需求文档，也能继承相关架构约束和工程规范。

---

## 6. 横向传播：从一个模块迁移到相邻模块

横向传播是指一个模块中的经验被抽象后迁移到其他模块。

### 示例

用户模块实现了统一的审计日志机制。后来订单模块、支付模块也需要类似能力。

这时知识不一定直接晋升到 L0，而是可以从用户模块 README 横向迁移到 L2 的“审计日志能力说明”，再由其他模块引用。

### 适合横向传播的知识类型

| 知识类型 | 示例 |
|---|---|
| 可复用设计模式 | 审计日志、状态机、策略模式 |
| 通用错误处理 | 重试、降级、补偿 |
| 通用测试策略 | 契约测试、回归测试、边界用例 |
| 通用 Agent 约束 | 某类代码生成前必须检查哪些文档 |

## 7. 知识生命周期

建议为每条知识定义生命周期状态。

```text
Candidate 候选
  ↓
Verified 已验证
  ↓
Adopted 已采纳
  ↓
Canonical 权威知识
  ↓
Deprecated 已废弃
  ↓
Archived 已归档
```

### 7.1 状态说明

| 状态 | 含义 |
|---|---|
| Candidate | Agent 或人工识别出的候选知识，尚未评审 |
| Verified | 已被代码、测试、评审或线上结果验证 |
| Adopted | 已被团队采纳，进入正式文档 |
| Canonical | 成为权威知识，后续需求和 Agent 必须遵守 |
| Deprecated | 已被新方案替代，不再推荐使用 |
| Archived | 已归档，仅用于历史追溯 |

---

## 8. 知识晋升规则

一条知识满足以下条件时，可以考虑晋升：

| 条件 | 示例 |
|---|---|
| 多次出现 | 多个需求都遇到同一个设计约束 |
| 影响范围扩大 | 从单模块影响到多个模块 |
| 具有稳定性 | 不是临时 workaround |
| 有证据支撑 | 有代码、测试、线上结果、评审记录 |
| 对 AI Coding 有指导价值 | 能减少 Agent 误生成、重复踩坑 |

### 示例

PR 中反复出现以下要求：

```text
新增 API 必须同时补充 OpenAPI Schema、契约测试和错误码文档。
```

如果它一开始只是某个 PR 的要求，但后来成为所有 API 开发的共同规则，就应该晋升到 L0 或 L1 的工程规范中。

---

## 9. 知识降级规则

一条知识应该降级的情况包括：

| 情况 | 处理方式 |
|---|---|
| 只适用于某个需求 | 从规范层降级到需求文档 |
| 只适用于某个模块 | 从架构层降级到模块 README |
| 已被新方案替代 | 标记 Deprecated |
| 与现有权威知识冲突 | 进入冲突评审 |
| 只是一次性修复经验 | 归入排障记录或 PR 说明 |

### 示例

早期文档中写道：

```text
所有用户信息都从 UserService 查询。
```

后来系统引入 ProfileService。旧知识就不能继续作为架构层权威规则存在，而应该被标记为 Deprecated，并指向新规则。

---

## 10. 需求实现过程中如何识别知识

在一个需求从提出到上线的过程中，知识会不断产生。可以将其归为 8 类。

| 知识类型 | 识别信号 | 建议归档位置 |
|---|---|---|
| 业务规则 | “必须”“不能”“仅当”“除非” | L2 / L3 |
| 架构决策 | 为什么选 A 不选 B | L1 / L3 |
| 接口契约 | API、事件、字段、错误码变化 | L2 / L3 |
| 领域模型 | 实体、状态、关系、生命周期变化 | L1 / L2 |
| 实现约束 | 性能、并发、幂等、安全限制 | L2 / L3 / L4 |
| 测试知识 | 边界用例、回归场景、Mock 策略 | L3 / L4 |
| 排障知识 | Bug 根因、修复方式、避免复发 | L4，必要时晋升 |
| AI Coding 约束 | Agent 生成代码前必须参考的规则 | L0 / L1 / L2 |

---

## 11. 需求结项时的知识识别问题清单

每个需求或 PR 结项时，可以要求开发者或 Agent 回答以下问题：

```text
1. 这次需求新增或改变了哪些业务规则？
2. 是否改变了模块边界、接口契约或领域模型？
3. 是否产生了新的通用实现模式？
4. 是否踩到了已有文档没有覆盖的坑？
5. 哪些知识只适用于本需求，哪些知识可复用？
6. 哪些知识需要让后续 Agent 知道？
```

这组问题可以作为 AI Agent 的知识抽取提示词基础。

## 12. 使用知识卡片作为最小治理单元

不建议让 Agent 直接修改大段权威文档。更稳妥的方式是先将知识抽取为“知识卡片”。

### 12.1 知识卡片示例

```yaml
knowledge_id: KG-ORDER-STATE-001
title: 订单状态变更必须通过状态机
type: business_rule
source:
  - requirement: REQ-2026-051
  - pr: PR-1842
  - files:
      - order/domain/order_state_machine.py
      - order/service/order_service.py
scope:
  level: module
  module: order
status: candidate
confidence: 0.82
evidence:
  - 新增 OrderStateMachine
  - 测试覆盖非法状态流转
  - PR Review 中明确禁止直接更新 status
suggested_action: promote_to_L2
target_doc: docs/domain/order/state-machine.md
owner: order-team
reviewer: architect
ttl: 180d
```

### 12.2 知识卡片的价值

知识卡片具有以下价值：

1. Agent 可以先提交候选知识，而不是直接改权威文档；
2. 人可以评审、合并、驳回；
3. 后续可以追溯这条知识来自哪个需求、哪个 PR、哪些代码文件；
4. 可以计算知识热度、复用次数、过期风险；
5. 可以作为后续 AI Coding 上下文检索的最小单元。

## 13. Agent 在知识识别和流转中的分工

建议不要使用一个万能 Agent，而是拆分为多个职责清晰的 Agent。

---

## 14. Knowledge Collector Agent：知识采集

### 输入

```text
需求文档
技术方案
Chat 记录
Issue
PR 描述
Code Diff
测试变更
Review 评论
```

### 职责

识别以下内容：

```text
新增规则
接口变更
架构决策
测试策略
排障经验
AI Coding 约束
```

### 输出

```text
候选知识卡片
```

---

## 15. Knowledge Classifier Agent：知识分类

该 Agent 判断知识属于哪一类：

```text
业务规则
架构决策
接口契约
实现模式
测试知识
排障知识
Agent 约束
```

同时判断文档层级：

```text
L0 原则层
L1 架构层
L2 能力层
L3 需求层
L4 实现层
L5 临时层
```

### 示例

```text
“新增优惠券状态 expired”
→ 领域模型变更
→ L2 或 L3

“所有金额字段必须使用 Decimal，不允许使用 float”
→ 工程规范
→ L0 或 L1

“本次为了兼容旧数据临时保留 legacy_status”
→ 需求实现细节
→ L3 / L4，不应晋升
```

---

## 16. Knowledge Promotion Agent：晋升建议

该 Agent 负责判断知识是否应该晋升。

### 判断维度

```text
复用频率
影响范围
稳定性
风险等级
是否被测试验证
是否已有相似知识
是否与权威文档冲突
```

### 输出

```text
建议晋升到哪个文档
为什么晋升
证据是什么
需要谁评审
推荐修改内容
```

注意：该 Agent 不应直接自动晋升权威知识，而应输出可评审建议。

---

## 17. Conflict Detector Agent：冲突检测

冲突检测是文档治理中的关键环节。

该 Agent 需要检查：

```text
新知识是否与已有规范冲突
同一个概念是否有多个说法
接口字段是否文档和代码不一致
旧文档是否已经过期
需求文档是否违背架构原则
```

### 示例

已有规范：

```text
金额单位统一为分，使用 integer。
```

新需求文档：

```text
amount 字段使用元，类型 decimal。
```

Agent 应提示：

```text
发现接口契约冲突：

- 现有规范：金额单位为分，integer；
- 新需求：金额单位为元，decimal；
- 建议：进入架构 / 接口评审，不能直接合并。
```

## 18. Doc Writer Agent：文档更新

Doc Writer Agent 不应该自由发挥，而应该基于模板写入。

### 18.1 L2 领域规则文档模板

```markdown
## 规则名称

### 适用范围

### 规则描述

### 正例

### 反例

### 关联代码

### 关联测试

### 变更记录

### 适用于 AI Coding 的提示
```

### 18.2 AI Coding 提示示例

```markdown
### 适用于 AI Coding 的提示

当修改订单状态相关代码时，必须：

1. 优先查看 OrderStateMachine；
2. 不允许直接修改 order.status；
3. 新增状态时必须补充状态迁移测试；
4. 修改状态枚举时必须同步更新 API Schema。
```

这一节非常重要。因为文档不只是给人看的，也是给 Agent 看的。

---

## 19. Doc Governance Gate：PR 阶段文档治理门禁

在 PR 阶段，可以加入一个 Agent Gate。

它检查：

```text
代码是否变更了接口
代码是否变更了领域模型
代码是否变更了配置
代码是否变更了数据库表
代码是否新增了错误码
代码是否修改了核心流程
是否同步更新了对应文档
```

### 示例提示

```text
本 PR 修改了 order_status 字段和 OrderStateMachine，
但未更新以下文档：

- docs/domain/order/state-machine.md
- docs/api/order.md

建议补充：

- 新增状态 paid_failed；
- paid → paid_failed 的触发条件；
- 相关错误码；
- 回滚和重试策略。
```

这比单纯要求“请更新文档”更有效。

## 20. 可落地的 Agent 工作流

可以将需求实现过程设计为以下流程：

```text
需求创建
  ↓
Agent 读取需求，抽取初始知识卡片
  ↓
Agent 检索相关 L0 / L1 / L2 文档，生成上下文包
  ↓
开发 / AI Coding 实现
  ↓
Agent 对比 Code Diff、测试、PR 评论，识别新增知识
  ↓
Agent 生成知识卡片和文档变更建议
  ↓
Conflict Agent 检查冲突
  ↓
Reviewer 确认晋升 / 保留 / 降级 / 废弃
  ↓
Doc Writer Agent 更新文档
  ↓
知识索引刷新，供后续 Agent 使用
```

## 21. AI Coding 上下文包

“上下文包”是 AI Coding 中非常关键的一环。

不要让 AI Coding Agent 盲目读取全量文档，而是根据任务生成精准上下文。

### 示例

本次任务：

```text
修改订单取消逻辑。
```

Agent 应提供：

```text
1. 订单领域模型；
2. 订单状态机规则；
3. 取消订单相关 API；
4. 幂等处理规范；
5. 历史相关需求和 Bug；
6. 当前模块测试策略。
```

这样 AI Coding 才能基于正确知识生成代码。

---

## 22. 不是所有知识都应该进入高层文档

可以使用以下矩阵判断知识去向。

| 知识价值 | 稳定性 | 去向 |
|---|---|---|
| 高价值，高稳定 | 晋升到 L0 / L1 / L2 |
| 高价值，低稳定 | 保留在 L3，观察后再晋升 |
| 低价值，高稳定 | 放在模块 README 或实现说明 |
| 低价值，低稳定 | 保留在 PR / Issue，不进入正式文档 |

### 示例

```text
这次因为历史数据脏，需要临时兼容 null 状态。
```

这不应该晋升为领域规则，只应记录在需求实现说明或迁移说明中。

但如果多次出现类似问题，可以归纳并晋升为：

```text
历史数据兼容策略：领域服务必须对 legacy 状态做显式转换，不允许业务代码散落处理。
```

这才是有治理价值的知识。

## 23. AI Coding 场景下优先治理的知识

以下知识会直接影响 Agent 生成代码的正确性，应优先治理：

```text
1. 模块边界：哪些代码能改，哪些不能改。
2. 领域规则：业务状态、金额、权限、流程约束。
3. 接口契约：字段、错误码、兼容性要求。
4. 工程规范：日志、异常、事务、幂等、重试、安全。
5. 测试要求：必须补哪些测试，哪些测试不能删。
6. 历史坑点：过去 Agent 或开发者容易犯的错误。
7. 禁止事项：不能调用的 API、不能绕过的服务、不能直接改的字段。
```

尤其要把“禁止事项”显式写出来。

AI Agent 对“应该怎么做”通常比较敏感，但如果没有明确约束，很容易生成看似合理但破坏架构的代码。

## 24. 知识治理评分

可以为每条知识计算一个治理评分：

```text
Knowledge Score =
复用度 × 影响范围 × 风险等级 × 稳定性 × 验证强度
```

### 动作建议

| 分数 | 动作 |
|---|---|
| 80 - 100 | 建议晋升到 L0 / L1 / L2 |
| 60 - 79 | 保留为候选，进入评审 |
| 40 - 59 | 保留在需求或模块文档 |
| 0 - 39 | 不进入正式文档，只归档 |

Agent 可以自动计算初始分数，但最终晋升到权威文档最好仍然需要人工评审。

---

## 25. 最小可行落地方案

可以先从以下 7 件事开始：

1. 建立文档分层：L0 到 L5，每类文档明确边界；
2. 引入知识卡片：所有新知识先成为候选卡片；
3. 在 PR 阶段运行 Agent：识别代码变更带来的知识变化；
4. 建立晋升规则：复用、稳定、影响范围、验证证据；
5. 设置文档 Owner：每个 L1 / L2 文档必须有负责人；
6. 建立冲突检测：防止新需求破坏已有架构知识；
7. 让文档反哺 AI Coding：每次编码前生成任务相关上下文包。

## 26. 简化闭环

```text
需求产生知识
  ↓
Agent 抽取知识卡片
  ↓
Agent 分类：规则 / 决策 / 契约 / 模式 / 坑点
  ↓
Agent 判断层级：L3 / L2 / L1 / L0
  ↓
人审：晋升 / 保留 / 降级 / 废弃
  ↓
Doc Agent 更新文档
  ↓
索引进入 AI Coding 上下文
  ↓
后续 Agent 编码时自动引用
```

## 27. 总结

AI Coding 文档治理的核心，是把需求实现过程中产生的隐性知识，从聊天、代码、PR、Review 和测试中抽取出来，变成可评审、可追溯、可复用、可被 Agent 消费的结构化知识资产。

一套成熟的机制应该包含：

- 清晰的文档层级；
- 明确的知识生命周期；
- 可评审的知识卡片；
- 自动化的知识识别 Agent；
- 文档冲突检测 Agent；
- PR 阶段的文档治理门禁；
- 面向 AI Coding 的上下文包生成机制。

最终目标是形成一个闭环：

```text
需求实现产生知识
  ↓
知识沉淀进文档体系
  ↓
文档体系反哺 AI Coding
  ↓
AI Coding 产生新的知识
  ↓
继续治理和演进
```

这样，文档不再只是项目的说明材料，而会成为 AI Coding 系统中的核心知识基础设施。
